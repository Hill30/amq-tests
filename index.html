Case Study: Enterprise Integration with Mobile Applications

Introduction

Integrating enterprise hosted application with application deployed on mobile devices is challenging as it requires providing enterprise-level quality services over unstable networks with inconsistent bandwidth and security. We created this study to share knowledge obtain from real-life business deployment, to facilitate feedback and sharing of best practices, to put some extra emphasis on this and similar architectural solutions for open source products development to improve products’ stability and performance.

Tech Objectives

  1. Provide reliable communications with distributes network of mobile devices,
  2. Seamless integration of broad mobile network with internal messaging system.
  3. Support for most standard enterprise communication patterns:
    • Broadcasting and pair-to-pair communications.
    • Load-balancing and fault-tolerance
    • Message routing, filtering, etc...
    • Messages' expiration and escalation
    • Reliable message delivery
    • Secure communication.
  4. Utilize standard and best practices for communications with growing network of mobile devices:
    • Standard-based communication protocol
    • Supported on Android and iOS,
    • Non-reliable network connections,
    • Energy efficient communication patterns
    • Performance related:
      • Max concurrently connected mobile devices/clients – 10,000.
      • Max communication traffic: 1,000 messages 1Mb total per device per day.
      • The latency of the message delivery for connected client – 60 seconds.
  5. Minimized maintenance of messages
    • Undelivered messages to mobile devices should expire in 48 hours.
    • Dormant client subscriptions should be removed with all messages cleared in 10 days.

Methodology and Assumptions

  • Test environment will be used verify business objectives.
  • Testing will be done first for single broker and then for load-balanced deployment.
  • Test client will be created to simulate real-life mobile devices.
  • Network disconnect will be emulated by standard server firewall turn off-on.
  • Required solution’s security will be provided by standard products’ functionality

Architecture Highlights

  • Internal system components implemented as services
  • User-facing applications are iOS and Android applications.
  • Administrative and internal power user applications are web-based.
  • Message Broker will provide support for enterprise Integration patterns (point-to-point, broadcasting, routing, user permissions) for both internal and external applications.
  • User authentication/authorization will be integrated with existing client’s security system.
  • Application is responsible for data consistency and validation, deals with message duplicates.

Components/Products

Internal services implemented in .Net following existing company’s standards.

Message Broker Selection

Considered:
  • Rabbit MQ
  • Tibco EMS
  • Active MQ

  • Zero MQ

    Active MQ was selected as message broker for the following reasons:

    • Proven, mature, enterprise message broker platform
    • Support for great variety of communication protocols.
    • Available integration adapters with other proprietary and open-source message brokers.
    • Available support from third parties and original development team.
    • Build-in integration with Apache Camel for configurable message routing.

Communication Protocol Selection

Considered:
  • Internet browser-based
  • Plain HTTP-based
  • Plain TCP

  • MQTT

    MQTT communication protocol was selected for communication with mobile devices. Reasons:

    1. Built for message-based communications with mobile devices.
    2. Available integration with Active MQ and other major message brokers.
    3. Integration client libraries available for both iOS and Android
    4. Widely accepted by industry for “Internet of Things” type of integrations.
    5. Standard protocol (OASIS).
    6. Binary compatibility/interoperability to minimize vendor dependency.
    7. Proven standard protocol (current version 3.1.1)
    8. For Android - Paho client-side library development supported by IBM.
    9. For iOS – native open source libraryhttps://github.com/ckrey/MQTT-Client-Framework.

Active MQ – MQTT Integration Highlights

ActiveMQ Configuration.

  1. Added/enabled MQTT adapter to Active MQ configuration.
  2. Increased max connections to 10,000
  3. Changed communication protocol to NIO (greatly reduces number of active threads for large number connections).
  4. Enabled more memory for the broker - 4GB (this is maximum using default wrapper that comes with ActiveMQ installation package).
  5. Enabled thread pool.
  6. Added transport level security – SSL. Changes extracted from config: … …

MQTT Client

  1. Active MQ topics, default communication channel will be used for connection with Active MQ broker.
  2. Messages sent to MQTT will use durable (to store-and-publish messages for disconnected subscribers) topics.
  3. MQTT will be sending message to non-durable topic (default MQTT adapter implementation in ActiveMQ).

Active MQ Topic - Queue Routing

MQTT – Active MQ adapter receives incoming messages in non-durable topic. The broker is configured to use Camel to route all messages from this to non-durable topic to a queue. This provides durability for messages to survive broker restart. It also provides support for built-in load-balanced message processing.

Active MQ - Server-Side Integration

Server side components implemented using .Net framework and integrated with Active MQ using NMS implementation library [http://activemq.apache.org/nms/].

Server – Client communication uses AMQ durable topics which are mapped to MQTT channels by MQTT AMQ adapter. To make topic/messages durable (persistent to file) application service will be setting messages’ quality of service flag QoS to 1 or 2.

Server specifies time-to-live for client messages to trigger expiration for non-delivered messages in the broker.

Client-Server communication uses a combination of AMQ non-durable topic where messages received from MQTT adapter and queues to provide durability and load-balanced processing. Apache Camel (with very basic, XML mapping is used to transfer messages from input topic to application processing queue.

Target Deployment Configuration

ActiveMQ

  1. Individual MQTT user mapped to dedicated durable topic.
  2. Messages sent to durable topic have expiration timeout.
  3. Incoming from MQTT client messages needs to be routed to a queue for durability and load-balanced processing.
  4. Enabled Camel with pre-set XML-based configuration for topic to queue message routing.
  5. Messages persisted in KahaDB (default) store.
  6. Load Balancing and Failover choice: Network of brokers with duplex inter-broker communication.
    Pros:
    • Standard, most common configuration
    • No need for shared date store (KahaDB) instance.
      Cons:
    • Combination with durable topics results in MQTT client receiving duplicate messages switching between brokers.
    • Incoming from MQTT messages need to be filtered from replication to avoid duplicates in re-routed queue.
    • Every broker in Network of Broker configuration will use its own persistent store.
